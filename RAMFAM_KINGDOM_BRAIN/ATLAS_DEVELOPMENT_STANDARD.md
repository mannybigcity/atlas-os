# ATLAS DEVELOPMENT STANDARD
## Version 1.0
### July 4, 2026

---

# PURPOSE

This document defines how all Atlas development work must be performed.

Atlas is no longer built by vague prompts.

Atlas is built through approved governing documents, focused implementation tasks, acceptance criteria, testing, and disciplined cost control.

---

# DEVELOPMENT LAW

Every Atlas development task must honor:

**Massive Action.**

**Maximum Effort.**

**Minimal Money.**

This means:

Move fast.

Do excellent work.

Avoid waste.

Do not rebuild what already works.

Do not spend money or AI tokens unnecessarily.

---

# SOURCE OF TRUTH

Atlas development must follow these approved documents:

- ATLAS_CONSTITUTION_V2.md
- KNOWLEDGE_ENGINE_SPEC.md
- EXECUTIVE_HANDBOOK.md
- EXECUTIVE_OFFICE_SPEC.md
- ATLAS_MASTER_IMPLEMENTATION_PLAN.md

If implementation conflicts with the governing documents, the documents win.

---

# PAPERCLIP RULES

Paperclip is the builder, not the architect.

Before beginning work, Paperclip must:

1. Read only the governing documents relevant to the task.
2. Inspect existing project files before creating new ones.
3. Identify what already works.
4. Avoid rebuilding existing working features.
5. Estimate scope and expected AI cost.
6. Stop if the task appears larger than approved.

---

# COST CONTROL RULES

Every task must:

- Prefer deterministic code before AI.
- Use the cheapest capable model.
- Avoid GPT-5.5 unless Manny explicitly approves.
- Load only relevant files.
- Never load all markdown files by default.
- Reuse existing code and assets when possible.
- Stop and ask for approval if cost risk increases.

---

# TASK FORMAT

Every Paperclip task must include:

- Task ID
- Objective
- Scope
- Files allowed to modify
- Files not allowed to modify
- Acceptance criteria
- Test plan
- Rollback plan
- Budget guardrail
- Implementation summary

---

# ACCEPTANCE CRITERIA RULE

A task is not done because code was written.

A task is done only when:

- Acceptance criteria are satisfied.
- Existing features still work.
- Tests pass or manual verification is documented.
- Changes are summarized clearly.
- Next recommended task is identified.

---

# MEMORY RULE

Atlas must never solve recurring problems by sending larger prompts.

Atlas solves recurring problems by:

- Updating specifications.
- Improving memory.
- Improving tests.
- Improving workflows.
- Improving architecture.

Think once.

Build many.

---

# SCOPE CONTROL

If a task discovers a bigger issue, Paperclip must not expand the task silently.

Paperclip must report:

- What was discovered.
- Why it matters.
- Recommended next task.
- Whether current task can still be completed safely.

---

# HUMAN APPROVAL

No task may:

- Spend money.
- Deploy to production.
- Publish content.
- Contact customers.
- Modify critical credentials.
- Delete major files.

without Manny's approval.

---

# STANDARD COMPLETION REPORT

Every completed task must end with:

- What changed
- Files changed
- Tests run
- What was not changed
- Any risks
- Recommended next step
- Estimated actual cost

---

# FINAL PRINCIPLE

Atlas shall grow through disciplined execution, not chaotic expansion.

Every task must move Atlas closer to becoming the world's best Virtual Executive Operating System for entrepreneurs.

Massive Action.

Maximum Effort.

Minimal Money.

Helping Families Create More Saturdays.
