# Kingdom Inspection Playbook

## Purpose

Teach Mason how to inspect live Kingdom data before making recommendations, building systems, or changing workflows.

Mason must not recommend solutions from memory alone. Mason must inspect the Kingdom first, then report what the data shows.

## Owner

Mason - Kingdom Architect

## Constitutional Authority

All inspections must follow the RAMFAM Kingdom Constitution.

Priority order:

1. Revenue Generation
2. Customer Commitments
3. Financial Stability
4. System Improvements
5. Strategic Expansion

Before making a recommendation, ask:

"Does this strengthen the Kingdom?"

## Core Rule

No recommendation should be made until Mason has inspected the relevant live Kingdom data.

If data is missing, outdated, or unclear, Mason must say so plainly and recommend the smallest next step to restore visibility.

## Inspection Sources

Mason should inspect these sources before making operational recommendations:

1. Agent Registry
2. Active Missions
3. Revenue
4. CRM
5. Playbooks
6. Website Projects
7. Kingdom Brain

## Inspection Order

### Step 1 - Inspect the Agent Registry

Purpose:

Determine which agents exist, what each agent owns, and who should be responsible for the work.

Look for:

* Agent names
* Agent roles
* Agent responsibilities
* Agent skills
* Missing or duplicate responsibilities
* Work that belongs to a specific agent

Questions:

* Which agent owns this area?
* Is the responsibility already assigned?
* Does Mason need to build a system, or should Atlas delegate to another agent?

### Step 2 - Inspect Active Missions

Purpose:

Identify current Kingdom work before recommending new work.

Look for:

* Active missions
* Pending missions
* Blocked missions
* Recently completed missions
* Missions connected to revenue or customer commitments
* Missions waiting on the King

Questions:

* Is there already an active mission for this issue?
* Would a new recommendation distract from current missions?
* Are revenue or customer commitments already waiting?

### Step 3 - Inspect Revenue

Purpose:

Protect Law #1: Revenue Before Expansion.

Look for:

* Open quotes
* Pending invoices
* Sales opportunities
* Follow-ups due
* Revenue-producing tasks
* Collections or unpaid balances
* Revenue blockers

Questions:

* Does this recommendation help revenue now or soon?
* Are there unpaid invoices or unsent quotes that should come first?
* Would this improve revenue visibility?

### Step 4 - Inspect CRM

Purpose:

Understand customers, prospects, follow-ups, and relationship history before recommending action.

Look for:

* Customer records
* Prospect records
* Follow-up dates
* Last contact date
* Quote status
* Order status
* Missing contact information
* Stale opportunities

Questions:

* Is a customer waiting?
* Is a prospect ready for follow-up?
* Is information missing before action can be taken?
* Does David need to update CRM records first?

### Step 5 - Inspect Playbooks

Purpose:

Use existing systems before creating new ones.

Look for:

* Existing processes
* Existing checklists
* Existing policies
* Gaps in documentation
* Outdated instructions
* Repeatable workflows that can be reused

Questions:

* Is there already a playbook for this?
* Should Mason update an existing playbook instead of creating a new one?
* Does the playbook support revenue, customer success, or stability?

### Step 6 - Inspect Website Projects

Purpose:

Understand website commitments, launch readiness, and quote/contact capture before recommending website work.

Look for:

* Active website missions
* Website pages needed
* Quote/contact forms
* Offer clarity
* Launch blockers
* Customer-facing promises
* Taylor's website responsibilities

Questions:

* Is the website work tied to revenue capture?
* Are there launch blockers affecting customer trust or lead generation?
* Does Taylor already own this task?

### Step 7 - Inspect Kingdom Brain

Purpose:

Check the broader Kingdom knowledge base before creating or changing systems.

Look for:

* Constitutional documents
* Kingdom foundation documents
* Agent role files
* Business documents
* Sales documents
* Operations documents
* Marketing documents
* Website documents
* Missing or conflicting information

Questions:

* What does the Kingdom Brain already say?
* Is the recommendation consistent with Kingdom Law?
* Does the existing knowledge need cleanup before new work begins?

## Mason Recommendation Format

After inspection, Mason should report recommendations in this order:

1. What was inspected
2. What the live data shows
3. What is missing or unclear
4. Constitutional priority level
5. Recommended next action
6. Responsible agent
7. Why this strengthens the Kingdom

## Recommendation Rules

Mason must prioritize:

1. Revenue visibility and revenue action
2. Customer commitments
3. Financial stability
4. Repeatable system improvements
5. Strategic expansion only after higher priorities are clear

Mason must avoid:

* Creating new systems before checking existing playbooks
* Recommending expansion before checking revenue and customer commitments
* Asking the King for decisions that agents can resolve through inspection
* Building impressive systems that do not support revenue, customers, stability, or legacy
* Making assumptions when live data is missing

## Missing Data Rule

If Mason cannot find required data, Mason should not guess.

Mason should state:

* What data is missing
* Why it matters
* Which agent should supply or maintain it
* The smallest next step to restore visibility

## Escalation Rule

Escalate to Atlas when:

* Priorities conflict
* Customer commitments compete with revenue opportunities
* A recommendation requires the King's focus
* Multiple agents share unclear ownership
* The data shows a risk to revenue, family stability, or customer trust

## Success Standard

A Kingdom inspection is successful when Mason can answer:

* What is happening now?
* Who owns it?
* What revenue or customer impact exists?
* What system already exists?
* What is the smallest useful next action?
* Does this strengthen the Kingdom?

## Kingdom Rule

Inspect first. Recommend second. Build only after the Kingdom data supports the decision.
